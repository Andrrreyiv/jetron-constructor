# Архитектура проекта: конструкторы кастомизации Jetron Sport

Версия: 1.0 (2026-07-09). Основано на `ТЗ.md` (разделы 1-11) + 4 партии голосовых.
Цель документа: зафиксировать техническое устройство до написания кода, чтобы не переделывать.

---

## 0. Ключевые архитектурные решения (кратко)

| # | Решение | Почему |
|---|---------|--------|
| A1 | **WordPress-плагин** `jetron-constructor`, а не просто статический HTML | Нужны: хранилище конфигурации (цены/шрифты/сетки/формы), админ-панель, корзина WooCommerce, почта. Всё это — серверная часть WP. |
| A2 | **Fabric.js v6 + чистый JS (ES-модули)**, без тяжёлого фреймворка | Клиент жалуется, что чужие конструкторы «зависают». Минимум абстракций = скорость + лёгкая встройка в WP. |
| A3 | **Конфиг-driven**: конструктор ничего не хардкодит, читает JSON-конфиг | Цены, шрифты, сетки, формы, зоны — редактируются из админки без правки кода (требование 10.5). |
| A4 | **Два входа, одно ядро**: `core/` общий, `uniform/` и `tshirt/` — режимы | Общая база (canvas, палитра, загрузка, текст, экспорт), различается логика размещения (ТЗ п.1). |
| A5 | **Разработка офлайн-первой**: статический стенд с mock-конфигом → потом обёртка в плагин | Можно собрать и протестировать форму без доступа к сайту (сайт под антиботом). Ускоряет старт. |
| A6 | **Зоны в процентах от canvas**, не в пикселях/см | Покупателю см не показываем; на любой ширине (мобилка) всё масштабируется пропорционально (9.2). |
| A7 | **Векторизация — офлайн-пайплайн исполнителя**, вне продукта | Отдельная платная услуга (11.4), не часть кода конструктора. |

**Развилка, требующая подтверждения клиента:** WooCommerce да/нет (риск 8.1). Архитектура готова к обоим путям, по умолчанию проектируем под WooCommerce add-to-cart, с fallback «заявка на почту».

## 2. Frontend: структура модулей

```
src/
  core/
    ConfigLoader.js     # получает конфиг (REST или inject через wp_localize), нормализует
    CanvasManager.js    # обёртка Fabric.js: слои (подложка формы / зоны / элементы), рендер
    ZoneManager.js      # зоны = bounding-box'ы; clamp внутрь, привязка, показ/скрытие
    TextTool.js         # поля фамилия/номер, шрифт, цвет, авто-подгонка под зону
    ImageTool.js        # загрузка, размещение, масштаб, лимиты
    FontManager.js      # динамический FontFace, кириллический fallback → РПЛ
    PriceCalculator.js  # движок правил: сумма позиций − скидки, live-пересчёт
    ExportManager.js    # превью PNG (toDataURL) + design JSON (состояние дизайна)
    UploadManager.js    # лимит 15 МБ, web-сжатие, вызов remove-bg
    RemoveBg.js         # клиентское удаление белого фона (порог по alpha)
    DesignState.js      # состояние дизайна (add/move/remove), undo
  uniform/
    UniformConstructor.js  # перёд+спина одновременно, фикс. зоны, плечо сбоку
  tshirt/
    TshirtConstructor.js   # перёд/зад со сдвигом, свободное размещение, лимит A3
    LibraryPanel.js        # библиотека готовых картинок по темам (векторные)
  ui/
    ColorPalette.js     # кружочки цветов
    ModelCarousel.js    # карусель форм выбранного цвета (внизу)
    Toolbar.js          # кнопки: надпись, загрузить, скачать, оформить
    OrderModal.js       # детское/взрослое → размер → гетры → итог → в корзину
    SizeTable.js        # таблица размеров из конфига (детская 3 колонки / взрослая RU)
    MobileLayout.js     # компактная мобильная раскладка (обязательна, ТЗ 7)
  entry-uniform.js      # точка входа конструктора формы
  entry-tshirt.js       # точка входа конструктора футболки
```

Сборка: **esbuild/vite** → один минифицированный бандл на конструктор (`uniform.bundle.js`, `tshirt.bundle.js`) + CSS. Плагин их enqueue'ит.

## 3. Модель данных (конфигурация)

Хранится в WP (options + Custom Post Types), отдаётся фронту как JSON.

### 3.1 Форма/модель — CPT `jetron_form`
```json
{
  "id": "champion-black",
  "name": "Champion",
  "color": "#000000",
  "colorName": "Чёрный",
  "images": { "front": "url", "back": "url", "shoulder": "url|null" },
  "zones": [ /* см 3.2 */ ]
}
```

### 3.2 Зона нанесения
```json
{
  "key": "back-name",
  "label": "Фамилия",
  "view": "back",          // front | back | shoulder
  "type": "name",          // name | number | logo | text
  "box": { "x": 0.36, "y": 0.18, "w": 0.28, "h": 0.075 }, // доли canvas
  "price": 600,            // или 0
  "included": false,       // включено в стоимость формы (шорты)
  "maxLines": 2,
  "align": "top-center"
}
```
Координаты — доли (0..1) от размеров подложки. Реальные см (28×7,5 и т.д. из `зоны-формы.md`) конвертируются в доли один раз при разметке модели.

### 3.3 Шрифт — CPT/опция `jetron_font`
```json
{ "id": "rpl", "name": "РПЛ", "fileUrl": "url", "cyrillic": true, "default": true }
```

### 3.4 Прайс и скидки — опция `jetron_prices` (JSON)
```json
{
  "forms": { "adult": 1280, "kid": 1090, "gaiters": 450 },
  "placements": {
    "back-name-number": 600, "chest-logo-small": 300, "chest-logo-big": 300,
    "chest-number": 300, "back-logo": 300, "shoulder-logo": 300,
    "shorts-number": 0, "shorts-logo": 0
  },
  "baseFee": 0,
  "discounts": {
    "jetronChest": 0.05, "jetronBack": 0.05,
    "bulk": { "threshold": 20, "freeChestLogoSmall": true }
  }
}
```

### 3.5 Размерные сетки — опция `jetron_sizes` (JSON) или связка с таблицами WP
```json
{
  "kids":   [ { "label": "4XS", "height": "120-128", "age": "5-6" }, ... ],
  "adults": [ { "order": "S (ASIA)", "ru": "44 RU (XS)" }, ... ]
}
```
Показываем: детская — 3 колонки (бирка/рост/возраст); взрослая — колонка «Российский размер».

## 4. Ключевые технические решения

### 4.1 Авто-подгонка текста под зону (номер/фамилия)
- Fabric `Textbox`, `fontSize` вычисляется динамически.
- 1 символ номера → высота = высота зоны.
- 2+ символов → если ширина превышает зону, уменьшаем `fontSize` до вписывания по ширине, центрируем (align top при необходимости, 9.2).
- Текст нельзя вытащить за рамку зоны (ZoneManager.clamp).

### 4.2 Одновременный показ перёд+спина, плечо сбоку
- CanvasManager рисует две подложки рядом (перёд слева, спина справа), спина приоритетна (2.2).
- Плечо НЕ на теле: при активации зоны «логотип на плече» — отдельный маленький превью-холст сбоку с лого/текстом (9.3).
- Гетры на макете не показываем (форма без шорт, 2.2).

### 4.3 Свободное размещение (футболка) + лимит A3
- Свободное перетаскивание/масштаб; верхний предел — A3 (~30×40 см) в пропорции canvas; ниже — маленький логотип (3.2).
- Перёд по умолчанию; клик «зад» → перёд уезжает влево, справа появляется спина (3.1).

### 4.4 Кнопка «удалить фон» (11.5, опционально)
- `RemoveBg.js`: читаем пиксели через canvas, порог по «белому» (RGB близко к 255) с edge-flood-fill, чтобы не выбить белый внутри лого → ставим alpha=0.
- Клиентский, без сервера. rembg (серверный) — только если порога мало; отложено.

### 4.5 Производительность (боль клиента «зависают»)
- Ограничить число объектов Fabric, `renderOnAddRemove=false` + ручной batch-render, debounce на drag.
- UploadManager сжимает большие фото в web (max-размер, WEBP/JPEG) перед добавлением (4.1 ТЗ, голос 11:14).
- Library-картинки — lazy-load тонкими полосками по темам.
- Отдельная мобильная раскладка, тач-таргеты под палец (7).

### 4.6 Шрифты и кириллица
- FontManager грузит `FontFace` динамически из конфига.
- Кириллические шрифты добавлены в список (11.2), РПЛ по умолчанию.
- Латинский шрифт + ввод кириллицы → fallback РПЛ + подсказка (10.3).
- Дугу НЕ реализуем (11.3).

## 5. Оформление заказа (два пути)

### 5.1 Путь WooCommerce (если плагин активен)
1. `OrderModal`: детское/взрослое → размер (из `jetron_sizes`) → предложить гетры → итог (PriceCalculator).
2. «В корзину»:
   - Файлы пользователя заранее загружены через `/upload` → в Media Library, получены URL.
   - `add-to-cart` базового товара-конструктора с cart item data: `design_json`, `files[]`, `preview_png`, `computed_price`, `size`, `gaiters`.
   - Хуки: `woocommerce_add_cart_item_data`, `woocommerce_get_item_data` (показ в корзине), кастомная цена через `woocommerce_before_calculate_totals`.
3. На заказ (`woocommerce_new_order`/`thankyou`) → письмо на **grc2@bk.ru** с превью + ссылками на файлы; всё лежит в meta заказа (6).

### 5.2 Fallback (нет WooCommerce)
- POST `/order` → сохраняем submission (CPT `jetron_order`) + письмо на grc2@bk.ru с превью и файлами.

### 5.3 «Скачать изображение»
- `ExportManager.toPNG()` → download. Без заказа (12). Печатный высокого разрешения НЕ нужен.

## 6. Админ-панель (WP admin, требование 10.5)

Меню «Jetron Конструктор»:
- **Цены** — форма редактирования `jetron_prices` (формы, гетры, позиции, скидки, бонус от 20 компл.).
- **Шрифты** — загрузка .otf/.ttf, флаг «кириллица», выбор дефолтного.
- **Размерные сетки** — редактирование детской/взрослой таблиц (или связка с существующими).
- **Формы/модели** — список; «Добавить форму»: загрузка мокапов (перёд/спина/плечо) + **Zone Mapper**.

### Zone Mapper — самый сложный узел
- Fabric.js в админке: поверх загруженного мокапа админ рисует/тянет прямоугольники зон, назначает `type`/`price`.
- На выходе — координаты в долях (0..1) → сохраняются в `zones[]` модели.
- Без него привязку зон под каждую модель пришлось бы делать вручную в коде. Это оправдывает доп. объём (риск 8.7).

## 7. Векторизация — офлайн-пайплайн исполнителя (вне продукта)

- Инструмент: `vtracer` (цветные → SVG) / `potrace` (моно). Себестоимость ~0.
- Вход: JPG от клиента → выход: SVG/векторные ассеты → идут в Library футболки (3.3) и/или отдаются клиенту.
- Ценообразование — услуга, не код: 150 ₽/шт, пакет 50 = 6000 ₽, сложные от 500 ₽ (11.4).
- В репозитории: отдельный `tools/vectorize/` скрипт, не попадает в бандл сайта.

## 8. План реализации по фазам

| Фаза | Содержание | Результат / проверка |
|------|-----------|----------------------|
| **0. Стенд** | Статический harness + mock `config.json` + локальные шрифты | Запускается локально, без WP |
| **1. Конструктор формы** (приоритет) | core + uniform: палитра, карусель, зоны, текст авто-fit, загрузка лого, плечо сбоку, цена live, экспорт PNG/JSON, мобилка | Собирается рабочая форма по топ-3 моделям; проверка desktop+mobile |
| **2. Конструктор футболки** | tshirt: плотность, перёд/зад сдвиг, свободное размещение A3, Library-панель | Рабочая футболка |
| **3. WP-плагин** | shortcode/блок, REST `/config`, Admin: цены/шрифты/сетки | Конфиг из админки читается фронтом |
| **4. Заказ** | `/upload`, WooCommerce add-to-cart (или fallback), письмо grc2@bk.ru | Заказ доходит с файлами+превью |
| **5. Zone Mapper** | Админ-разметчик зон на мокапе | Клиент сам добавляет форму |
| **6. Опции + полиш** | remove-bg, скидки, QA мобилки, оптимизация | Не зависает, тач-friendly |

Векторизация (офлайн) — параллельно, по мере присылки картинок.

**Блокеры до фазы 1-4:** (1) подтверждение WooCommerce; (2) файл топ-3 моделей от клиента; (3) размеченные зоны по этим моделям.

## 9. Границы объёма (важно для цены)

Входит в базовые 25 000 ₽: два конструктора (формы + футболки) с нанесением, ценой, заказом.
**Сверх базовой цены / риск** (проговорить с клиентом): WooCommerce-интеграция глубиной (8.1), стандартизация фото-мокапов (8.2), полноценная админ-панель + Zone Mapper (8.7). Векторизация и remove-bg — отдельные потоки.

## 10. Критика плана (self-review) и усиление

> Разбор по шаблону «оцени идею до старта». Всё сверено с `ТЗ.md`. MVP-нарезка v1/v2 — это **фазирование, а не урезание ТЗ**: финальная сдача = полный объём ТЗ.

### 10.1 Что сильного и реально сработает
- **Config-driven + WP-плагин** — соответствует требованию 10.5, даёт клиенту редактируемость без кода.
- **Fabric.js + vanilla, зоны в долях canvas** — прямой ответ на боль «конструкторы зависают» (17) и на пропорциональную мобилку (9.2, 7).
- **Офлайн-стенд первым (A5)** — снимает блокер антибота, старт не ждёт доступов.
- **Ядро + два режима (A4)** — минимум дублирования между формой (2) и футболкой (3).
- **3 модели → тиражирование** (ТЗ 2.1 / голос) — правильно, не универсальный шаблон сразу.

### 10.2 Что слабое и может провалиться
- **WooCommerce-развилка не разрешена (8.1)** — это фундамент всего заказа (ТЗ 6), а мы «под оба». До подтверждения клиента — риск переделки cart-слоя. Самый большой риск.
- **Zone Mapper недооценён (10.5, 8.7)** — это мини-редактор в админке, самый дорогой узел; в базовых 25к он «спрятан». На старте зоны топ-3 моделей заводим **вручную в JSON**, Mapper — в v2.
- **Кастомная цена в WooCommerce** капризна: пересчёт через `woocommerce_before_calculate_totals`, легко получить дыру, если доверять цене с фронта.
- **Мусор в Media Library** от брошенных корзин (файлы грузятся до заказа) — нужен TTL/чистка, в ТЗ нет, добавляем.
- **Стандартизация мокапов (8.2)** переложена на клиента — если фото разноракурсные, зоны «поплывут». Риск качества, вне нашего контроля.
- **remove-bg порогом (11.5)** — на градиентном/не-белом фоне работает плохо. Не обещать «идеально», позиционировать как «для белого фона».

### 10.3 Что, скорее всего, не учли (добавить в реализацию)
- **Безопасность (критично):** серверная валидация итоговой цены (не доверять фронту); проверка типа/размера файла; **санитизация загруженного SVG** (SVG = XSS-вектор в WP). Не было в разделах 5-6 — фиксируем как обязательное.
- **Снапшот заказа:** design JSON + **зафиксированная цена и версия зон** сохраняются в заказ, чтобы правки конфига админом не «ломали» старые заказы (стык с 5.1, 6).
- **152-ФЗ:** если на форме заказа собираем контакты — отдельный непредзаполненный чекбокс согласия на ПДн (скилл `russian-content-compliance`).
- **Лицензии клубных шрифтов (10.3)** — правовой риск при коммерческой продаже; предупредить клиента, риск не наш.
- **Логирование ошибок конструктора** в проде — чтобы ловить реальные «зависания», а не обещать.
- **Кириллица в латинском шрифте** — fallback РПЛ (10.3) требует живого теста, не только кода.

### 10.4 Обязательное vs лишнее на старте (в рамках ТЗ)
- **v1 (обязательно, конструктор формы, ТЗ 2 + 6 + 7):** палитра + карусель, фикс. зоны, фамилия+номер с авто-fit, загрузка лого, плечо сбоку, live-цена, экспорт PNG+JSON, оформление заказа (минимум email-fallback на grc2@bk.ru), мобилка.
- **v2 (по ТЗ, но позже):** Zone Mapper (10.5) — на старте JSON; полная админ-панель (10.5) — на старте минимум опций; конструктор футболки (ТЗ 3, этап 2); библиотека картинок (3.3, этап 2); remove-bg (11.5, по да/нет клиента); WooCommerce add-to-cart (если подтвердят — иначе email).
- **Не делаем (прямо по ТЗ):** проверка качества картинок (4), дуга (11.3), печатный высококач. макет (12).

### 10.5 Ошибки новичков в таких продуктах (и как избегаем)
- Универсальный конструктор под все модели сразу → тонут. **У нас: 3 модели, потом тираж.**
- Доверяют цену с фронта → подмена. **У нас: серверный пересчёт (10.3).**
